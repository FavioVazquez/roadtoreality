## Common Pitfalls

### Tone mismatch: writing a case study instead of a chronicle
**What goes wrong:** The writer treats the section as a project summary ("In phase 1 we did X, in phase 2 we did Y") rather than a first-person account that conveys what it *felt like* to work with learnship on this project.
**Why:** The existing episodio4.md is written in a confessional, direct register ("Quiero ser honesto sobre cómo llegué a construirlo", "me quedó pendiente"). It names emotions, surprises, and friction. A case-study bullet list would clash immediately.
**How to avoid:** Match the existing voice precisely. Use first-person ("construyendo", "me encontré", "lo que no esperaba"). Lead with an observation or realization, not a summary.

### Describing learnship mechanics that are already explained elsewhere in the article
**What goes wrong:** Repeating explanations of DECISIONS.md format, the Phase Loop steps, or how `/verify-work` works — all of which are already covered in the sections preceding `## El proyecto de ejemplo`.
**Why:** The preceding sections (`## El Phase Loop`, `## DECISIONS.md: el registro de por qué`, `## El aprendizaje integrado`) already cover the mechanics. The project section must apply those mechanics to a real case, not re-explain them.
**How to avoid:** Reference the mechanics by name ("el Phase Loop", "el CONTEXT.md de fase", "el UAT conversacional") and immediately pivot to what happened specifically in this project.

### Getting the project facts wrong
**What goes wrong:** Saying "13 stops" when the truth is nuanced (13 with full simulations, 37 with rich stub pages + teaser animations, total 50 stops defined in stops.json). Or calling it "phases 1-5" without noting v1.0 vs v2.0 distinction.
**Why:** The article presents Favio as someone who builds things for real and reports on them honestly. Inaccuracies undermine that.
**How to avoid:** Use the precise language from STATE.md:
  - "13 simulaciones interactivas completas" covering Eras 1–2 (stops 001–013)
  - "37 páginas stub con animaciones de teaser" — not blank placeholders, real animated canvas pages
  - "50 paradas en total" — all defined in stops.json
  - v1.0 milestone = phases 01–05, completed 2026-03-21
  - v2.0 phase 06 = UX polish (KaTeX, search, stub pages, fixes), also completed 2026-03-21
  - Current position: ready for phase 07 (Open Graph images)

### Treating the UAT issues as minor footnotes
**What goes wrong:** Mentioning "we found and fixed some bugs" without specifics.
**Why:** The UAT issues in phase 06 are exactly the kind of concrete evidence that makes the "behind the scenes" framing credible. They show learnship's UAT loop working as designed: issue found → diagnosed → fix plan written → executed.
**How to avoid:** Name the specific issues:
  - **027 Planck blackbody:** Animation was technically running (rAF loop firing) but motion was imperceptible — ±0.2 opacity glow only. Diagnosis: not a code bug but a design problem. Fix: replaced with clearly dynamic motion (curve build-in, pulsing peak, temperature sweep).
  - **040 Nuclear Fission:** Chain reaction looped correctly but fired in milliseconds — too fast to see. Root cause: daughter nuclei spawned only 16px from parent (dist = nuc.r × 0.8), so neutrons traveled ≈6 frames to reach them. Fix: increased spawn distance, slowed neutron speed, added per-nucleus delay timer.
  - **Search discoverability:** Cmd/Ctrl+K worked but no visible affordance. Fix: added search trigger button with platform-correct keyboard hint (⌘K / Ctrl+K) to all 51 page headers.

### Treating DECISIONS.md as just a list of technical choices
**What goes wrong:** Summarizing decisions as "we chose X over Y" without conveying *why this matters* in an agentic workflow.
**Why:** The article's argument is that DECISIONS.md prevents "arqueología de decisiones" — the cost of arriving at a code choice weeks later with no memory of why it exists. The section should show this working in practice.
**How to avoid:** Pick 2–3 decisions that best illustrate the problem DECISIONS.md solves:
  - **DEC-006 (RK4 vs Euler):** Not obvious from code alone which sims use which integrator or why. The decision doc records that RK4 is for energy-conserving systems (orbits, pendulums) and Euler is sufficient for short-duration one-directional sims.
  - **DEC-010 (Noise-based galaxy background):** The initial approach (stacked radial gradients) was rejected — a future agent encountering the canvas code would assume it was the "natural" choice. DEC-010 explains the rejection and why per-pixel noise rendering was chosen instead.
  - **DEC-002 (No build pipeline):** Simple and easy to second-guess. The doc records the explicit rejection of Vite, Eleventy, Webpack and why.

### Making the section too short or too long
**What goes wrong:** Either a brief 3-paragraph summary (too thin given the placeholder's promise of "transcripciones reales") or an exhaustive phase-by-phase log (too heavy for an episodio article).
**Why:** The placeholder text (lines 152–159) previewed this as a rich section with real transcripts and architectural decisions. It sits between `## El Phase Loop` and `## Paso a paso`, so it must be substantial enough to serve as the concrete payoff to the abstract model.
**How to avoid:** Aim for 800–1200 words in the final Spanish prose. Structure around 3–4 narrative beats: (1) the project and why it was chosen, (2) what the Phase Loop actually produced (CONTEXT.md decisions, DECISIONS.md entries, UAT issues), (3) one or two specific moments that could only happen with learnship in place (the 027 diagnosis, the DECISIONS.md preventing context drift across 6 phases), (4) current status and what's next.
